## 1. 登录与账号

### 1.1 登录返回 401「用户名或密码错误」

| 排查 | 操作 |
|---|---|
| 输错了 | 让用户重输；注意大小写 |
| 账号被店主停用 | 店主在「员工管理」查 `is_active`，启用即可 |
| 密码改了忘了 | 店主在「员工管理」找到该员工 → 「重置密码」生成新临时密码 |
| 数据库里 hash 坏了 | 平台运维找 `users` 表，让开发用 BCrypt 重新生成 hash |

### 1.2 登录成功但看不到任何菜单

JWT 里的 `role` claim 为空或角色不在路由白名单。

| 排查 | 操作 |
|---|---|
| 浏览器 F12 → Application → Local Storage 看 token | 解码 JWT 看 `role` 是否正确 |
| 该用户表里 `role` 错了 | 店主在「员工管理」改角色 |
| `tenant_id` 为空（非 PlatformAdmin） | 平台运维查租户绑定，DB 改 `users.tenant_id` |

### 1.3 平台管理员账号能登但看不到「按摩店租户」

- 角色不是 PlatformAdmin（`role=0`），其他角色看不到平台菜单 → 改角色
- 路由 Guard 拦了 → 检查 `roles` meta，刷新页面

### 1.4 默认 admin@123 改不了

首次登录后**必须先去「个人设置 → 修改密码」**——直接关页面再来还是默认密码（系统不强制 force change，依赖人）。

## 2. 收银与订单

### 2.1 「请为所有项目指派技师」

购物车里某一行没选技师。下拉里选一个即可。

### 2.2 「实收金额不足」

现金支付时输入金额 < 应付。检查实收输入框。

### 2.3 「会员卡余额不足或未关联会员」

- 卡上没钱 → 改用其他支付方式
- 没选会员 → 选会员再结账

### 2.4 结账后小票没打出来但订单已生成

打印机故障**不会阻塞结账**——订单已经在系统里。处理：

1. 「订单流水 → 选订单 → 重新打印」
2. 排查打印机（见 [3](#3-打印机--钱箱)）

### 2.5 反结账失败

- 已经过了反结账时限（默认 24 小时）→ 店长加权限可手工放宽
- 已封盘工资里包含了这一单提成 → 反结账不动工资，提成会从下月调整里扣

### 2.6 加钟后总价不对

加钟单的"原服务时长"会保留，**新增"加钟服务"行**。如果你看到的是总价异常，检查会员折扣、套餐扣次是否叠加正确。

### 2.7 转钟后两个技师的提成都不对

提成切分按"服务时长占比"。如果客人主诉是"换人之前的部分别给原技师"，店长可以在「订单流水 → 提成调整」手工 0 计。

### 2.8 计时房计费偏高 / 偏低

- 检查"开始时间"是不是错了（误点 → 在「房间管理」里修正）
- 检查"小时单价"是不是配错（店长在「房间管理 → 该房 → 编辑」改）

### 2.9 团购券核销提示「已使用」

券码已经被核销过。两种情况：

1. 客人重复给同一张券 → 当面验证（美团 / 点评商家后台）
2. 系统重复核销 → 查 `voucher_redemptions` 表是否有两条；这是 bug，上报开发

## 3. 打印机 / 钱箱

### 3.1 「测试打印小票」点了没动静 / 报错

弹窗会显示失败原因。按原因排：

| 错误信息 | 处理 |
|---|---|
| 找不到串口 COMx | 确认 Windows 设备管理器里 COMx 是否存在；可能 USB-串口线没插好 |
| 端口被占用 | 关闭其他用 COM 的程序（如串口调试助手）|
| 网络不通 | `ping 打印机IP`；网线松没接好 / 打印机断电 |
| 写超时 | 打印机缓冲区满 / 卡纸 → 重启打印机 |
| 编码错误 | `printer.codePage` 是 GBK / 936 之一 |

### 3.2 小票出来是乱码

- `codePage` 改成 GBK
- 仍乱：试 GB18030
- 国产偏门打印机用厂商自己的代码页 → 联系厂商技术

### 3.3 钱箱不弹

- `openDrawerOnCash` 是不是 `true`
- 钱箱与打印机间的 RJ-11 线是不是松
- 钱箱里的钩子卡住了 → 手动开一次

### 3.4 切纸不齐 / 不切

- `cutPaper` 是不是 `true`
- 刀片磨损 → 更换刀片或整机
- 部分廉价打印机无切纸功能 → 改成 `false` 走半切或手撕

### 3.5 走纸越界

- `charsPerLine` 配置错了：58mm 用 32 / 80mm 用 48
- 改完重启 CS 端

## 4. 客显

### 4.1 客显黑屏

- `customerDisplay.enabled = true`
- 电源插紧
- 串口号正确
- 仍黑：用串口调试助手发 `ESC @`（0x1B 0x40）测试是否亮

### 4.2 客显有内容但不刷新

- 业务代码报错被压在日志里 → 查 CS 端日志 `客显写入失败`
- 串口被其他程序占用

### 4.3 客显文字乱码

- `codePage` GBK
- 部分老式客显只支持 ASCII + 自定义字库 → 改用纯 ASCII 描述（在工单里向厂商问字库格式）

## 5. 来电盒

### 5.1 来电不播报

- `callerId.enabled = true`
- 串口号正确
- 来电盒电话线接对（线序 RJ11）

### 5.2 来电盒接进来但无数据

- 用串口调试助手监听 COMx，让一个手机拨入店里座机
- 看是否有 `NMBR=xxx` 字符
- 没有 → 来电盒型号不输出 ASCII 模式（找厂商切换 modem 模式）
- 有但 CS 端不响应 → 日志查 `来电串口异常`

### 5.3 号码识别错位

- 波特率不对（一般 1200，老旧 modem 用 110）
- 来电盒输出格式跟正则不匹配 → 上报开发调整 `NumberLine` 正则

## 6. 磁条读卡器

### 6.1 刷卡没反应

- `cardReader.enabled = true`
- 焦点在收银台界面（其他界面会语音提示"请先切换到收银台再刷卡"）
- 串口号正确

### 6.2 调出的会员不对

- 卡号读了部分位 → 检查波特率
- Track2 数据被读卡器自定义裁剪了 → 用串口调试助手看原始数据，对比 `ExtractCardNumber` 逻辑

### 6.3 我用的是 USB HID 键盘式读卡器（自动打字）

- **不要**启用 `cardReader.enabled` —— 启用后会试图打开 COM 端口失败
- HID 模式下，刷卡数据会被任意聚焦的输入框吃掉 → 焦点放在「会员搜索」框即可
- 缺点：盲人收银员切焦点容易冲突；建议 RS232 模式

## 7. 会员卡 / 套餐 / 团购券

### 7.1 会员卡余额跟客人记的不一样

- 「会员详情 → 操作历史」逐项对账
- 仍有偏差：查 `audit_logs`；最常见是上一收银员当时输错了实充金额

### 7.2 会员等级显示错了

- 等级是按"累计消费"算的，到了上一档但页面没刷新 → F5 刷新
- 规则改了但未生效 → 后台规则改变只影响新结算，已存档不变

### 7.3 计次卡用了但次数没扣

- 服务项目和套餐的"适用服务"对不上 → 店主在「会员套餐」里编辑套餐的适用范围

### 7.4 团购券扫码报「无效」

- 券码输错（少位）
- 该券非本店渠道（如美团但你输的是点评码）
- 已过期 → 让客人改用现金 / 微信

## 8. 微信支付 / 支付宝

### 8.1 客人付了钱但订单还显示 Pending

- 等 30 秒：回调有轻微延迟
- 仍 Pending：让客人发支付截图核对金额
- 在「订单流水 → 该单 → 详情」看 `payment_orders` 状态：
  - `Pending`：回调没到 → 见 8.2
  - `Paid` 但订单状态 Pending：回调到了但订单状态没同步 → 看 API 日志，可能是事务回滚

### 8.2 微信支付回调收不到

| 排查 | 操作 |
|---|---|
| 回调 URL 在公网可达？ | `curl -X POST https://api.your-domain.com/api/callback/wechat` 应返回 401/400 而不是 timeout |
| HTTPS 证书有效？ | `curl -v https://api.your-domain.com/health` |
| 回调路径在中间件白名单？ | 在 `TenantStatusMiddleware` 中应包含 `/api/callback/*`（默认就是） |
| 商户后台回调 URL 配对了？ | 商户平台 → 产品中心 → 开发配置 → 异步通知 URL |
| 防火墙挡了微信的 IP？ | 看 Nginx access log，应有微信侧的 POST 请求 |

### 8.3 微信支付回调 200 但订单没变

- 看 API 日志找 `out_trade_no=xxx` 处理记录：
  - `AmountMismatch`：金额单位错。微信发"分"，本系统用"元"。重发时确保转换正确
  - `SignatureFailed`：APIv3 密钥或证书错。比对生产 ENV 里的密钥
  - `OrderNotFound`：商户订单号在本系统找不到。可能是测试与生产环境串了

### 8.4 重发回调

不要直接改 DB。商户后台「订单查询」找到这笔，点「补发回调」。本系统回调幂等，重发安全。

### 8.5 续费失败但钱扣了

平台端「按摩店租户 → 查看订单」找到 Pending 单：

1. 看 `out_trade_no` 在微信商户后台查实际状态：
   - 微信说成功 → 平台后台「线下激活」补续期（同时让运维查回调日志找根因）
   - 微信说失败 → 24 小时内退款给客人

## 9. 微信订阅消息

### 9.1 客人没收到通知

按"绑定 → 授权 → 触发 → 发送"四步逐项查：

| 步骤 | 怎么确认 |
|---|---|
| 绑定 | 客人小程序「我的预约」里能看到自己的卡 → 绑定 OK |
| 授权 | 客人小程序里弹过订阅消息授权且点了同意 → 没弹过 / 拒了，引导重新进入小程序操作 |
| 触发 | 看后端日志，应该有 `Notify queued` 这一条 |
| 发送 | 看后端日志，应该有 `Notify sent` 或 `Notify failed: 43101 / 40001` |

### 9.2 错误码

| 错误码 | 含义 | 处理 |
|---|---|---|
| 43101 | 用户未授权 | 让用户在小程序里重新点触发授权的按钮（充值时会出 `wx.requestSubscribeMessage`） |
| 40001 | AppSecret 错 / access_token 过期 | 检查 `Notifications:WeChat:AppSecret`；系统会自动刷 token，过期是临时的 |
| 40036 / 40037 | 模板 ID 错 | 检查 `Notifications:WeChat:Templates:*` 配的 4 个 ID |

### 9.3 通知很久才到

- `Notifications:IntervalSeconds` 默认 300 秒（5 分钟）轮询一次
- 业务高峰时可临时调小到 60 秒——但微信侧有日发送上限，别为了快撞限额

### 9.4 测试时收不到

- 测试号 vs 正式号：`Notifications:WeChat:MiniProgramState` 控制（`trial` / `formal`）
- 模板必须**已审核通过**才能发；申请后到审核完成 1-7 天

## 10. 小程序

### 10.1 进不去小程序

- 检查微信版本（≥8.0）
- 检查小程序状态：管理员在公众平台看是否被处置 / 下架

### 10.2 提交预约报错"门店不可用"

`storefront` 接口拿不到门店——可能租户到期了（写请求被中间件拦），但读接口不应该被拦。检查后端日志。

### 10.3 显示空白

- 公众平台「开发设置 → 服务器域名 → request 合法域名」必须包含 API 域名
- 漏配会导致请求被微信侧拒绝，小程序内表现是页面空白
- 解决：去公众平台补上域名，**注意每月只能改 5 次**

### 10.4 读屏不读 / 朗读乱

- 关检查 VoiceOver/TalkBack 是否开
- 重启微信
- 小程序内 `aria-label` 缺失 → 上报开发（应有不存在的属于 bug）

### 10.5 字号太小

让用户改微信「我 → 设置 → 通用 → 字体大小」为「特大」。我们的 CSS 用 rpx 单位会随系统字号缩放。

## 11. 报表数据异常

### 11.1 营业额比手算少

| 原因 | 排查 |
|---|---|
| 反结账单未算入 | 默认就不算 |
| 小费未算入 | 默认就不算（小费单独显示） |
| 团购券抵扣后金额 | 报表按实际收入算，券抵的部分另计 |
| 时区错位 | 服务器时区应是 Asia/Shanghai；`select @@time_zone;` |

### 11.2 技师业绩跟工资对不上

- 业绩按"营业额"算（含会员折后实收）
- 工资按"提成基数 × 提成系数"算
- 两者本来就不一样——这是设计

### 11.3 报表慢

- 查询区间太大（如选了 5 年）→ 默认只该开放到 1 年
- 索引缺失：让运维 `EXPLAIN` 慢查询，按需加索引
- 数据库未清理：陈旧数据归档到独立表

### 11.4 当日数据没出现

- 报表是按"日结时间"还是"创建时间"？本系统按创建时间算
- 跨零点的订单可能两边都不在 → 看 `created_at`

## 12. 工资 / 提成

### 12.1 工资单跟技师自算的差很多

按顺序自查：

1. 「提成规则」对照：阶梯是不是配错？技师等级系数？
2. 「服务项目」对照：服务的"提成基数"对不对？
3. 「考勤」对照：满勤奖 / 加班 / 请假是不是漏算？
4. 「调整」对照：是不是上月有手工调整结到这月？

### 12.2 反结账后提成怎么处理

- 当月还未封盘：直接从该技师的当月提成里减
- 已封盘：本月不动，下月以「调整」项扣回

### 12.3 跨店调动后提成怎么算

- 调动当天技师做的单子，提成算"实际服务所在门店"
- 工资单按"原归属门店"出，但有"跨店调动收入"单列

### 12.4 工资单封盘了能改吗

不能在 UI 里改。误封了只能：

1. 联系运维改 `payroll_records.status` 回 Draft（**有审计风险**）
2. 或在下月「调整」里冲销

## 13. 到期软锁

### 13.1 店主反馈"我点啥都报 403 TenantExpired"

- 平台端查租户 `expire_at` < 当前
- 续费：让店主在「订阅状态」走支付 OR 平台「线下激活」
- 续完立即生效（无需重启系统）

### 13.2 到期了但 GET 接口能用

设计如此——读接口不锁，避免欠费时连数据都看不到。只锁写。

### 13.3 续费完了仍报 403

- 检查 `payment_orders` 是否 `Paid`、`tenant.expire_at` 是否更新
- 仍异常：浏览器 hard refresh / 退出重登（JWT 里没缓存到期信息，只 middleware 每次现查）

## 14. 系统性能 / 卡顿

### 14.1 API 响应慢

| 排查 | 操作 |
|---|---|
| 数据库连接打满 | MySQL `SHOW PROCESSLIST` 看 |
| 慢查询 | 看 `slow_query_log`；EXPLAIN 找无索引 |
| Redis 失联 | 看 API 启动日志，应连得上 Redis |
| CPU 满 | 服务器 `top`，看是不是 dotnet 进程 |

### 14.2 BS 端首屏白

- 浏览器 F12 → Network 看资源是不是 200
- Nginx `try_files $uri $uri/ /index.html` 配错会导致路由刷新 404 → 白
- API 接口 401 而前端没跳登录 → 检查 `index.ts` Guard

### 14.3 CS 端打开慢

- .NET 8 Desktop Runtime 没装 → 安装
- 首次启动有 JIT 编译，正常；持续慢可能是网络慢（连 API）

### 14.4 数据量大后报表卡

- 后端报表查询应有时间分片
- 数据库表过大：分表 / 归档；找开发评估

## 15. 紧急联系流程

按严重度分三档：

### P0 全店瘫痪（紧急）

现象：所有人登不上 / 所有结账失败 / 数据库挂

1. 店长立即电话平台运维（电话见合同）
2. 同时启动应急流程：
   - 收银走手工纸单
   - 技师按平时流程接客
   - 客人多的情况先开单不结账，恢复后补
3. 运维到位后：
   - 看 `/health` 状态
   - 看服务器 `top` + MySQL 状态
   - 查 Nginx / API 日志最近 10 分钟错误

### P1 部分功能挂（影响部分人）

现象：某个菜单进不去 / 某种支付不能用 / 报表打不开

1. 店长在工作群里报：现象、复现步骤、影响人数
2. 平台运维 1-2 小时内介入
3. 同时记录哪些客人被影响，后续补偿

### P2 偶发 / 不阻塞

现象：偶尔 UI 卡顿 / 个别按钮不灵 / 不影响主流程
