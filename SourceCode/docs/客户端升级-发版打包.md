# 客户端在线升级 · 发版与打包

CS 端（WPF）与移动端（安卓）已支持启动自动检测升级、强制更新。本文是**发新版**时的操作清单。

## 机制概览

- 后端平台级表 `app_versions`（不做租户隔离），平台端「客户端版本发布」菜单维护。
- 客户端启动匿名调 `GET /api/app-versions/check?platform=Cs|Android&version=<当前版本>`，
  返回是否有更新、是否强制（当前版本 < 最低支持版本即强制）、下载地址、更新日志。
- CS 端下载 `setup.exe` 自动安装；移动端应用内下载 `.apk` 唤起系统安装器。

> 注意鸡生蛋：升级功能本身要先随各端发一版到用户手里，之后发布的新版本才能被检测到。

---

## 发版总流程（每次发新版）

1. **改版本号**
   - CS：`backend/src/MassageSaas.Cs/MassageSaas.Cs.csproj` 的 `<Version>`（如 `1.1.0`）。
   - 安卓：`frontend/apps/mobile/android/app/build.gradle` 的 `versionName`（给人看）与 `versionCode`（每次+1，整数）。
2. **打安装包**（见下方各端命令）。
3. **上传安装包**到 dl 服务器（如 `https://dl.qiyuanrj.com/...`），拿到下载直链。
4. **平台端发布**：「客户端版本发布」→ 发布新版本，选平台、填版本号、下载地址、更新日志；
   要强制更新就填「最低支持版本」；勾选启用。（可选填安装包大小/ SHA256，CS 端会校验。）
5. 用户下次启动客户端即检测到。

## CS 端（WPF）打包 setup.exe

### 1. 发布自包含程序（win-x64）

```powershell
# 在 backend 目录
dotnet publish src/MassageSaas.Cs/MassageSaas.Cs.csproj `
  -c Release -r win-x64 --self-contained true `
  -o src/MassageSaas.Cs/bin/Release/net8.0-windows/publish-cs
```

### 2. 用 Inno Setup 打成 setup.exe（推荐）

安装 [Inno Setup](https://jrsoftware.org/isdl.php)，写一个 `.iss` 脚本指向上面的 `publish-cs` 目录，
`AppVersion` 与 csproj 的 `<Version>` 保持一致，输出 `齐源按摩-setup-<版本>.exe`。
关键：安装目录、开始菜单/桌面快捷方式、覆盖安装时先结束正在运行的进程。

> 升级时 CS 端会下载该 setup.exe 并以 `UseShellExecute` 启动、随后退出自身，
> 安装包需支持**覆盖安装**（同一 AppId）。如需算 SHA256 填到平台端：
> `Get-FileHash 齐源按摩-setup-1.1.0.exe -Algorithm SHA256`。

## 移动端（安卓）打包 APK

### 1. 构建前端并同步到原生工程（关键）

```powershell
# 在 frontend 目录
npm run build:mobile
npm -w shop-mobile exec cap sync android
```

> **务必 `cap sync android`**：本功能新增了原生插件
> `@capacitor/filesystem` + `@capacitor-community/file-opener`，不 sync 不会编进 APK，
> 升级下载/安装会直接失效。

### 2. 用 Android Studio 或 Gradle 打签名 release APK

```powershell
# 在 frontend/apps/mobile/android 目录（需已配置签名 keystore）
./gradlew assembleRelease
# 产物：app/build/outputs/apk/release/app-release.apk
```

签名 keystore 与 `signingConfigs` 在 `android/app/build.gradle` 配置（沿用现有签名，
保证覆盖安装不被「签名不一致」拦截）。把 `app-release.apk` 改名为带版本号后上传 dl。

### 3. 安装权限提示

- manifest 已加 `REQUEST_INSTALL_PACKAGES`；
- 用户首次安装时系统会要求允许「安装未知应用」，引导其在系统设置里授权即可。

### ⚠️ 原生工程不入库：重建 android/ 后需重新施加的配置

`frontend/apps/mobile/android/` 被 `.gitignore` 忽略（由 `cap add` 可重建、不入库）。
因此以下**原生侧改动只在本地磁盘**，一旦删掉 android/ 重新 `npx cap add android`，必须重新加回：

1. `android/app/src/main/AndroidManifest.xml` 加：
   ```xml
   <uses-permission android:name="android.permission.REQUEST_INSTALL_PACKAGES" />
   ```
2. 确认存在 FileProvider（`${applicationId}.fileprovider`）及 `res/xml/file_paths.xml` 含 `<cache-path>`
   （Capacitor 默认即带，用于把 Cache 里的 APK 经 FileProvider 共享给安装器）。
3. 配置 release 签名 `signingConfigs`（沿用既有 keystore，保证覆盖安装签名一致）。

> 插件依赖（filesystem / file-opener）在 `package.json` 里已入库，`cap sync` 会自动装回，无需手动。

## 后端发布（首次上线本功能必做）

升级相关接口与 `app_versions` 表是新增的，后端需：

1. 生产库执行 `发布包/migration-*.sql`（含 `AddAppVersion` 建表，幂等可重复执行）。
2. 部署后端增量包（含 `AppVersionsController`）。否则 `/api/app-versions/*` 会 404/500。

详见 `docs/IIS部署说明.md` 与发布脚本 `backend/publish-iis.ps1`。
